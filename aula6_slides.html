O Dilema do Chunking PlanoAula 6

1/5

Um único tamanho de chunk não serve bem a todos os tipos de consulta jurídica.

  • Query pontual: 'Qual o fundamento do voto do Min. Alexandre?' → precisa chunk pequeno (128 tok)
  • Query ampla: 'Quais as linhas argumentativas do acórdão?' → precisa contexto de múltiplas seções
  • Chunk pequeno: ✅ precisão | ❌ sem contexto para queries gerais
  • Chunk grande: ✅ contexto | ❌ ruído, degrada Context Precision no RAGAS
  • Solução: indexação multi-nível com hierarquia de granularidade

Parent-Child RetrieverAula 6

2/5

LlamaIndex indexa chunks pequenos (filhos) mas entrega chunks maiores (pais) ao LLM quando recuperados.

  • Child chunks (128 tok): indexados e buscados — alta precisão no retrieval
  • Parent chunks (512 tok): entregues ao LLM — contexto suficiente para geração
  • Desacopla o índice de busca do contexto de geração
  • Ratio recomendado: parent = 4× child; overlap no parent de 10%
Python 3.11+
from llama_index import VectorStoreIndex, StorageContext
from llama_index.node_parser import (
    HierarchicalNodeParser,
    get_leaf_nodes
)

parser = HierarchicalNodeParser.from_defaults(
    chunk_sizes=[512, 128]  # parent, child
)
nodes = parser.get_nodes_from_documents(documents)
leaf_nodes = get_leaf_nodes(nodes)

# Indexar apenas os filhos (leaf nodes)
storage = StorageContext.from_defaults()
storage.docstore.add_documents(nodes)  # armazena todos
index = VectorStoreIndex(leaf_nodes,   # indexa apenas filhos
                         storage_context=storage)

# Retriever que entrega o PAI ao recuperar o FILHO
from llama_index.retrievers import AutoMergingRetriever
retriever = AutoMergingRetriever(
    index.as_retriever(similarity_top_k=6), storage
)

RAPTOR — Sumarização em ÁrvoreAula 6

3/5

RAPTOR constrói uma árvore de sumarizações recursivas usando clustering UMAP + GMM.

  • Nível 0 (folhas): chunks originais do corpus
  • Clustering: UMAP reduz dim → GMM agrupa chunks semanticamente similares
  • Sumarização: LLM gera resumo de cada cluster → novos nós da árvore
  • Recursão: repete até restar 1 nó raiz (resumo total do corpus)
  • Retrieval: busca em todos os níveis simultaneamente

HyDE — Hypothetical Document EmbeddingsAula 6

4/5
HyDE resolve o gap semântico entre queries curtas e documentos longos usando um documento hipotético como ponte.
  • Problema: embedding de 'requisitos prisão preventiva' (5 palavras) ≠ embedding do acórdão (500 palavras)
  • Solução: LLM gera um 'documento hipotético' que responderia à query
  • O embedding do documento hipotético é muito mais próximo dos documentos reais
  • Melhora Recall@5 em 12-18% vs query embedding direta (Gao et al., 2022)
Python 3.11+
HYDE_PROMPT = """
Escreva um trecho de acórdão jurídico brasileiro que responderia
perfeitamente à seguinte pergunta. Use linguagem técnica-jurídica:

Pergunta: {query}

Trecho (apenas o texto, sem introdução):
"""

def hyde_retrieve(query: str, llm, retriever) -> list:
    # 1. Gerar documento hipotético
    prompt = HYDE_PROMPT.format(query=query)
    hyp_doc = llm.generate(prompt, max_tokens=300, temperature=0.7)

    # 2. Embedding do documento hipotético
    hyp_embedding = embedder.encode([hyp_doc])[0]

    # 3. Busca com embedding hipotético
    return retriever.retrieve_by_embedding(hyp_embedding, top_k=5)

Guia de Decisão: Qual Técnica Usar?Aula 6

5/5

Cada técnica se destaca em cenários específicos do domínio jurídico.

  • Parent-Child: documentos estruturados (acórdãos, laudos) com perguntas mistas pontuais+gerais
  • RAPTOR: corpus grande (>100 docs) com perguntas que exigem síntese cross-documento
  • HyDE: queries vagas ou sem correspondência lexical com os documentos
  • Combinação: HyDE para retrieval + Parent-Child para entrega de contexto ao LLM
