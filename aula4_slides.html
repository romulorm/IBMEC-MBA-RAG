Por que Busca Vetorial Pura Não BastaAula 4

1/7

No domínio jurídico, embeddings densos têm três fraquezas críticas que exigem busca híbrida.

  • Termos técnicos raros: 'art. 5º, LXIII, CF/88' — embeddings treinados em inglês codificam mal
  • Match exato de números: 'Lei 9.099/1995' — embeddings aproximam semanticamente, BM25 é exato
  • OOV (out-of-vocabulary): neologismos jurídicos, nomes de partes, datas de julgamento
  • Solução: Hybrid Search = BM25 (precisão lexical) + kNN (captura semântica)
BM25 (Lexical)
✅ Match exato de termos
✅ Números de lei
❌ Sem semântica
+
kNN (Semântico)
✅ Sinônimos e contexto
✅ Variação linguística
❌ Match exato ruim
=
Hybrid Search
✅ Precisão lexical
✅ Riqueza semântica
✅ RRF fusion

Arquitetura Híbrida no OpenSearchAula 4

2/7

O OpenSearch 3.x suporta busca híbrida nativamente com índice de campos duplos.

  • Campo textual (tipo 'text'): indexado com BM25 via Lucene, analisador 'portuguese'
  • Campo vetorial (knn_vector): dim=1024, HNSW, space_type=cosinesimil
  • Query híbrida: executa BM25 e kNN em paralelo, combina via Search Pipeline
  • Neural Search Plugin + ML Commons: registra modelo embedding no cluster
Python 3.11+
# Criação do índice híbrido
import json
from opensearchpy import OpenSearch

client = OpenSearch(hosts=[{"host": "localhost", "port": 9200}])

index_body = {
  "settings": {"index": {"knn": True}},
  "mappings": {
    "properties": {
      "conteudo": {"type": "text", "analyzer": "portuguese"},
      "embedding": {
        "type": "knn_vector",
        "dimension": 1024,
        "method": {
          "name": "hnsw",
          "space_type": "cosinesimil",
          "engine": "faiss"
        }
      }
    }
  }
}
client.indices.create("juridico_hybrid", body=index_body)

Reciprocal Rank Fusion (RRF)Aula 4

3/7

RRF combina rankings de múltiplas queries (BM25 + kNN) usando apenas a posição de cada documento — ignora totalmente os scores brutos. Esta é a chave para sua robustez.

  • Fórmula: RRF(d) = Σq  1 / (k + rankq(d)) — soma sobre todas as queries q
  • rank_constant (k): default 60 (Cormack et al. 2009). Menor → mais peso ao topo; Maior → suaviza diferenças
  • Por que funciona: BM25 (0–30) e kNN cosseno (−1..1) têm escalas incompatíveis. RRF ignora escala, considera só rank
  • Sintaxe OpenSearch 2.19+: processor score-ranker-processor (Neural Search plugin)
  • Aplicado em: queries do tipo hybrid via ?search_pipeline=...
OpenSearch 2.19+ · sintaxe oficial
# Search Pipeline RRF — sintaxe oficial
# Ref: opensearch.org/blog/introducing-reciprocal-rank-fusion-hybrid-search/
pipeline = {
  "description": "Post processor for hybrid RRF",
  "phase_results_processors": [{
    "score-ranker-processor": {
      "combination": {
        "technique": "rrf",
        "rank_constant": 60   # default; k ≥ 1
      }
    }
  }]
}
client.transport.perform_request(
    "PUT",
    "/_search/pipeline/rrf-pipeline",
    body=pipeline,
)

# Aplicar via parâmetro de query
client.search(
    index="juridico", body=hybrid_query,
    params={"search_pipeline": "rrf-pipeline"},
)

RRF vs Min-Max — Quando usar cada umAula 4

4/7

Duas estratégias distintas de rank fusion no OpenSearch — diferem na base matemática, no processor utilizado e nos casos ideais de uso.

✅ RRF
• Imune a escalas diferentes (BM25 vs kNN)
• Resistente a outliers (rank ordinal)
• ~1.6% mais rápido (BEIR oficial)
• Estável em dados dinâmicos
• Default seguro p/ alta vazão
✅ Min-Max
• Permite pesos custom por query
• Scores interpretáveis em [0, 1]
• ~+3.86% NDCG@10 média
• Ideal p/ distribuição conhecida
• Tunável p/ domínios específicos
🎯 Regra prática: comece com RRF (default seguro). Migre para Min-Max apenas se precisar priorizar explicitamente um ramo da busca (ex: 70% BM25 em domínio com terminologia técnica fixa).

Neural Sparse Search (SPLADE)Aula 4

5/7

Neural Sparse gera representações esparsas aprendidas — o melhor dos dois mundos: léxico + semântica.

  • SPLADE: Sparse Lexical and Expansion model — expande vocabulário semanticamente
  • Saída: vetor esparso com pesos por token (como TF-IDF mas aprendido por rede neural)
  • Vantagem sobre BM25: expande 'réu' para incluir 'acusado', 'indiciado' automaticamente
  • Indexado com tipo 'rank_features' no OpenSearch — eficiente para corpus jurídico
Python 3.11+
# Neural Sparse com modelo SPLADE no OpenSearch
from transformers import AutoTokenizer, AutoModelForMaskedLM
import torch

model_id = "opensearch-project/opensearch-neural-sparse-encoding-v1"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForMaskedLM.from_pretrained(model_id)

def encode_sparse(text: str) -> dict:
    inputs = tokenizer(text, return_tensors="pt", truncation=True)
    with torch.no_grad():
        output = model(**inputs).logits[0]
    values, indices = torch.topk(output.relu().log1p().max(0).values, 128)
    return {
        tokenizer.decode([i]): v.item()
        for i, v in zip(indices.tolist(), values.tolist())
    }

Contextual RetrievalAula 4

6/7
Técnica da Anthropic: adiciona contexto do documento completo a cada chunk antes de indexar.
  • Problema: chunks sem contexto — 'O réu foi condenado' não diz de qual processo
  • Solução: usa LLM para gerar 1-2 frases de contexto por chunk: 'Do Acórdão X, TJSP, 2023...'
  • Melhora Context Precision no RAGAS em até 15% (Anthropic, 2024)
  • Custo: 1 chamada LLM por chunk na ingestão — amortizado ao longo do tempo
Python 3.11+
CONTEXT_PROMPT = """
<document>
{full_document}
</document>

Aqui está um chunk deste documento:
<chunk>
{chunk_content}
</chunk>

Escreva 1-2 frases de contexto que situem este chunk no documento completo.
Responda APENAS com o contexto, sem explicações:
"""

def add_context(chunk: str, full_doc: str, llm) -> str:
    prompt = CONTEXT_PROMPT.format(
        full_document=full_doc[:4000],  # primeiras 4k chars
        chunk_content=chunk
    )
    context = llm.generate(prompt, max_tokens=100)
    return f"{context}\n\n{chunk}"  # prepend context

Métricas de Avaliação de BuscaAula 4

7/7

Três métricas para avaliar a qualidade do retrieval antes de chegar ao LLM.

  • MRR@K (Mean Reciprocal Rank): avalia onde a 1ª resposta relevante aparece no ranking
  • Recall@K: % dos documentos relevantes recuperados nos top-K resultados
  • NDCG@K: considera posição e grau de relevância — padrão em competições de IR
  • BM25 baseline → Dense → Hybrid → Hybrid+Contextual (progressão esperada)
